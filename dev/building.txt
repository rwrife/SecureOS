QUICK START (planned — gated on #408/#409/#410)
------------------------------------------------
  The compile-and-run flow below is the target experience. Until #408, #409,
  and #410 close, `cc` and launcher run wiring may be unavailable on-device.

  1. Compile the sample:
         cc /apps/dev/hello.c -o /apps/hello.bin
  2. Run it:
         hello
  3. You should see:
         hello from inside SecureOS

WRITING YOUR OWN APP (planned until #408/#409/#410)
----------------------------------------------------
  - Every app defines `int main(void)` (or `int main(int argc, char **argv)`).
  - Include the public API:  #include "secureos_api.h"
  - Print with:              os_console_write("...");
  - Return 0 from main on success; the value becomes the shell's $? .

  Example (mini.c):
      #include "secureos_api.h"
      int main(void) {
          os_console_write("my first app\n");
          return 0;
      }

  Build and run (planned while gates are open):
      cc /apps/dev/mini.c -o /apps/mini.bin
      mini

FILESYSTEM NAME LIMITS (important)
----------------------------------
  The SecureOS filesystem uses 8.3 names: at most 8 characters for the name
  and 3 for the extension. So:
      hello.c      OK
      mini.c       OK
      myutil.c     OK
      hello_world.c   TOO LONG  -> use hello.c
  Output binaries use the .bin extension (the OS wraps your compiled ELF in a
  signed-container format automatically).

TRUST / SIGNING
---------------
  Apps built inside the OS are NOT signed by the SecureOS master key (that
  private key never lives on the device). When you run a locally built app the
  system will prompt you to allow an unsigned binary the first time. Choose
  "Allow" to run it once, or "Always" to remember your choice for that binary.

  Audit trail: both allow and deny decisions are recorded as launch.granted /
  launch.denied events (see #554 for owner-kind audit-surface tracking).
