# Plan: Fix Ed25519 Verification on i386 Freestanding Target

**Date:** 2026-03-16  
**Status:** Pending  
**Priority:** High  

## Problem

Ed25519 `ed25519_verify()` fails on the i386 freestanding kernel target
(`clang --target=i386-unknown-none-elf -ffreestanding -m32`). Signing
(`ed25519_sign()`) completes without error but the resulting signature
does not pass `ed25519_verify()`.

### Diagnostic Output (from `about` command)

```
cert_parse:  ok
chain_valid: FAIL(5)     ← CERT_ERR_SIGNATURE_INVALID
payload_sig: FAIL(3)     ← ED25519_ERR_VERIFY_FAILED
```

Both cert signature (68-byte message) and payload signature (64-byte
SHA-512 hash) fail, confirming `ed25519_verify` itself is broken.

## Root Cause Hypothesis

The Ed25519 implementation uses `int64_t` field arithmetic and large
stack-allocated arrays. Possible causes:

1. **64-bit arithmetic on 32-bit target**: The compiler generates calls
   to `__divdi3`, `__moddi3`, etc. for 64-bit division on i386. In
   freestanding mode without compiler-rt, these may be missing or produce
   wrong results.

2. **Stack overflow**: `ed25519_verify` calls `sha512_hash` and performs
   double-scalar multiplication. Combined stack usage may exceed the
   kernel stack size, corrupting local variables.

3. **Compiler optimization bug**: `-O0` vs `-O2` may produce different
   results for 64-bit operations on 32-bit. The current build uses no
   explicit optimization flag.

## Investigation Steps

- [ ] Add `-O0` / `-O2` flag to the i386 clang invocation and compare
- [ ] Check if linking `libcompiler_rt` resolves 64-bit arithmetic
- [ ] Instrument `ed25519_verify` to print intermediate values and
      compare with a host-compiled reference
- [ ] Measure kernel stack usage with a canary pattern
- [ ] Test with a minimal sign+verify roundtrip at boot (before FS init)

## Mitigation (Current)

- `app_validate_codesign()` in `process.c` uses warn-and-allow for
  invalid signatures (development mode)
- `cert_set_root_public_key()` caches the derived root key to avoid
  mismatches between signing and verification key derivation

## Fix Options

1. **Fix the Ed25519 implementation**: Ensure all 64-bit ops work on
   32-bit (may require linking compiler-rt or rewriting field arithmetic
   to use only 32-bit types)
2. **Use a different signature algorithm**: Switch to a 32-bit-friendly
   algorithm (e.g., ECDSA with secp256k1 using 32-bit limbs)
3. **Pre-sign binaries on host**: Move signing to the build toolchain
   (host x86_64) and only verify on the kernel. This requires fixing
   verify only, or deferring verification entirely.

## Files Involved

- `kernel/crypto/ed25519.c` — sign/verify implementation
- `kernel/crypto/sha512.c` — SHA-512 (also uses uint64_t)
- `kernel/user/process.c` — `app_validate_codesign()` warn-and-allow
- `kernel/format/sof.c` — `sof_verify_signature()`
- `kernel/crypto/cert.c` — `cert_chain_validate()`, `cert_verify()`
